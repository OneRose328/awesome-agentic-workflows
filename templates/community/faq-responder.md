# FAQ Responder

## Purpose
Answer recurring support questions with concise, repeatable project guidance that reduces maintainer repetition.

## Trigger
Run when an issue or discussion asks a question that is common, documentable, and suitable for a short canonical answer.

## Instructions
1. Decide whether the question is a true FAQ, a one-off edge case, or something that really needs engineering triage instead of a stock answer.
2. Prefer answers that point to existing docs, canonical examples, or explicit repository policy rather than generating new unsupported guidance.
3. Answer in the smallest useful shape: one direct answer, one pointer, and one next step if needed.
4. Use safe outputs to post one public reply that either resolves the question or routes the person to the correct resource.
5. If the answer is uncertain, state what is known, state what is not, and defer cleanly instead of guessing.
6. Avoid sounding overconfident on security-sensitive, account-specific, or policy-sensitive questions in public.
7. If the question reveals a recurring docs gap, make the reply reusable so maintainers can later turn it into a doc update.

## Expected Output
- A short public answer that reduces repeated support work without sounding canned.
- A reply maintainers can reuse or turn into documentation later.

## Customization Notes
- Replace the sample guidance with links to your actual docs, setup guides, and support policy.
- Tune the response style to match whether your project prefers terse answers or slightly more guidance.
- Keep the reply compact so it feels like maintainer guidance, not autogenerated noise.
